## TL;DR

- AOS 是**单次触发式 Agent**，不是持续运行系统；多会话 = 变相并行，不同对话独立 Agent 实例通过文件系统交互
- 三种执行模式：**PLAN**（只读探索+生成计划）、**EXECUTE**（读写操作+工具调用）、**VERIFY**（新会话中只读验证+输出报告）
- VERIFY 模式**必须在新会话中发起**，审查报告写入 06_LOGS/，结果更新回 agent_tasks.json
- 多会话协同核心是 **agent_tasks.json**：任务生命周期 pending → in_progress → completed/failed，文件级锁+乐观并发控制防冲突
- 三层防死循环机制：工具层硬隔离（超时+最大重试）→ 推理层熔断（连续三次无效推理则中断）→ 规划层自修正（失败日志回传重新规划）
- Task 工具子代理使用规范：独立上下文，只返回最终结果，子代理描述中明确"只返回最终结果"
- 删除一个 Agent 往往比增加一个 Agent 更值钱——从最简单开始

## 核心概念

### 执行模型认知：单次触发式 Agent

AOS 的 Agent 不是持续运行的后台守护进程，而是**单次对话触发式**的执行单元。每次用户发起对话，Agent 实例被创建、执行任务、状态落盘、会话结束。不存在后台监听或常驻执行。

**关键约束**：
- 所有任务必须显式触发
- 所有状态必须写入磁盘，不能依赖运行时内存
- 每次会话启动时从磁盘读取状态，会话结束时状态必须落盘
- 多会话 = 变相并行：不同对话独立 Agent 实例，通过文件系统交互

> 引用依据：相关参考 明确指出 Agent/Subagent/Agent Teams 是"针对不同场景的专用工具"，选择原则是"用最简单能完成任务的模式"；相关参考 源码架构揭示 Tool-Use Loop 是 while(true) 循环，end_turn 即终止信号，Agent 天然是单次触发模式。

### 三种执行模式

| 模式 | 触发条件 | 行为约束 | 对应命令 | 权限级别 |
|------|----------|----------|----------|----------|
| PLAN | 复杂任务、架构决策 | 只读探索+生成计划，禁止写入 | `/spec` 或 `/plan` | 只读 |
| EXECUTE | 计划确认后、简单任务 | 读写操作+工具调用 | 默认模式 | 完整读写 |
| VERIFY | 任务完成前、用户要求审查 | 只读验证+输出报告 | **新会话中发起** | 只读 |

> 引用依据：相关参考 详解 Plan Mode 通过 EnterPlanMode/ExitPlanMode 两个工具实现，进入后权限降为只读，每5轮系统塞小纸条提醒还在 Plan Mode；相关参考 Harness Engineering 强调 Maker/Checker 分离是工程领域的基本常识。

### VERIFY 模式设计

VERIFY 模式的核心原则：**同一任务不能在同一会话中执行+验证**。这是 AOS 八条铁律之首——Maker/Checker 分离。

**触发条件**：
- EXECUTE 模式任务完成后，自动建议进入 VERIFY
- 用户主动要求审查
- Scheduled Loop 每轮执行完毕后

**执行流程**：
1. **必须在新会话中发起**——不能复用执行会话的上下文，避免自我评估偏乐观
2. 新会话启动后读取 agent_tasks.json，定位待验证任务
3. 只读验证：读取产出文件、运行测试（只读模式）、检查规范合规性
4. 输出审查报告至 `06_LOGS/verify_YYYYMMDD_HHMMSS_{task_id}.md`
5. 更新 agent_tasks.json 中对应任务的 result_summary 和 status

**审查报告格式**：

```markdown
# 审查报告：{task_id}

## 基本信息
- 任务描述：{description}
- 执行者：{claimed_by}
- 审查者：新会话 Agent 实例
- 审查时间：{timestamp}

## 验证项
| 验证项 | 结果 | 说明 |
|--------|------|------|
| 功能正确性 | ✅/❌ | ... |
| 规范合规性 | ✅/❌ | ... |
| 测试覆盖 | ✅/❌ | ... |
| 无副作用 | ✅/❌ | ... |

## 问题清单
1. ...

## 结论
- [ ] 通过
- [ ] 需修改后重审
- [ ] 需回滚
```

> 引用依据：相关参考 指出"模型评估自己输出时倾向给偏高评分"，原因是缺乏外部基准和自我一致性陷阱；相关参考 强调"验证不可外包——'完成了'是声明不是证明"。

## 运行时铁律（强制）

以下规则源于 Trae CODE 模式的实际能力边界，不可违反：

1. **用户是唯一调度器**：所有任务由用户触发，AOS 不假设 Agent 会自主行动或自主协调。Agent 不会主动读取 agent_tasks.json 并认领任务——必须由用户在新会话中明确指派
2. **状态更新是步骤的一部分**：写入 Skill/流程的强制步骤，不是可选的事后动作。Agent 不会"记得"回头更新文件，所以更新必须嵌入执行流程
3. **事前声明优于事后记录**：执行前声明"即将做什么"并写入文件，执行本身就是完成证明
4. **文件是唯一的跨会话通道**：不依赖 Agent 记忆，只依赖磁盘上的文件传递上下文
5. **禁止假设 Agent 会自觉更新**：任何设计如果依赖"Agent 执行完后主动更新状态"，则该设计不可靠

## 方法论 / 流程

### 多会话协同协议

#### agent_tasks.json 完整结构

```json
{
  "version": "1.0",
  "last_updated": "2026-06-21T16:00:00+08:00",
  "tasks": [
    {
      "task_id": "TASK_0001",
      "type": "feature|bugfix|refactor|verify|scheduled",
      "description": "实现用户认证模块的JWT刷新机制",
      "created_by": "session_abc123",
      "created_at": "2026-06-21T10:00:00+08:00",
      "target_files": [
        "01_PROJECTS/proj-auth/src/auth/jwt.rs",
        "01_PROJECTS/proj-auth/src/auth/refresh.rs"
      ],
      "status": "pending",
      "claimed_by": null,
      "claimed_at": null,
      "result_summary": null,
      "completed_at": null
    }
  ]
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| task_id | string | 唯一标识，格式 TASK_XXXX，自增 |
| type | string | 任务类型：feature/bugfix/refactor/verify/scheduled |
| description | string | 任务描述，需清晰无歧义 |
| created_by | string | 创建者会话ID |
| created_at | ISO8601 | 创建时间 |
| target_files | string[] | 涉及的目标文件路径列表 |
| status | string | 当前状态：pending/in_progress/completed/failed |
| claimed_by | string|null | 认领者会话ID |
| claimed_at | ISO8601|null | 认领时间 |
| result_summary | string|null | 执行结果摘要 |
| completed_at | ISO8601|null | 完成时间 |

#### 任务生命周期

```
pending → in_progress → completed
                      → failed
```

**状态转换规则**：

1. **pending → in_progress**：用户在新会话中明确指派任务（如："认领 agent_tasks.json 中的 TASK_0001 任务并执行"），Agent 按指令执行。认领时写入 claimed_by + claimed_at + status=in_progress。认领前必须检查 target_files 是否被其他任务锁定。
2. **in_progress → completed**：任务执行成功，写入 result_summary + completed_at + status=completed。
3. **in_progress → failed**：任务执行失败，写入 result_summary（含失败原因）+ completed_at + status=failed。失败任务可被重新认领。
4. **completed → pending**（重开）：验证不通过时，任务回退到 pending 状态，result_summary 追加验证失败原因。

#### 冲突处理：文件级锁 + 乐观并发控制

**文件级锁**：
- 每个任务的 target_files 声明了该任务要操作的文件范围
- Agent 认领任务前，必须检查 target_files 中的文件是否已被其他 in_progress 任务锁定
- 锁定粒度为文件级：不同任务操作不同文件可并行，操作相同文件必须串行

**乐观并发控制**：
- 读取 agent_tasks.json 时不加锁
- 写入前重新读取，比对版本号或 last_updated 时间戳
- 若写入前发现数据已被其他会话修改，则放弃本次写入，重新读取后重试
- 最多重试 3 次，3 次冲突后写入失败，任务状态回退

**死锁预防**：
- 任务认领时按 task_id 升序申请文件锁
- 不允许循环等待（A 等 B 的文件，B 等 A 的文件）
- 超时释放：claimed_at 超过 30 分钟且 status 仍为 in_progress 的任务，自动释放锁回到 pending

> 引用依据：相关参考 明确"文件级锁+乐观并发控制处理并发文件操作"；相关参考 源码中 StreamingToolExecutor 的 isConcurrencySafe 标记是工具级并行控制的参考。

### 三层防死循环机制

```
┌─ 第一层：工具层硬隔离 ──────────────────────────────────────┐
│ 每个工具调用设置超时时间和最大重试次数                         │
│ 超时时间：Read/Grep/Glob 30s，Edit/Write 60s，Bash 120s     │
│ 最大重试：3次，超过则工具调用失败                              │
│ 失败后不重试同类工具，切换策略或上报                           │
└──────────────────────────────────────────────────────────────┘
          ↓ 工具层未能阻止
┌─ 第二层：推理层熔断 ─────────────────────────────────────────┐
│ 追踪连续无效推理次数                                         │
│ 无效推理定义：工具调用返回结果未改变任务状态，且未推进目标进度   │
│ 连续3次无效推理 → 触发熔断，中断当前执行                     │
│ 熔断后输出简报：当前状态、已尝试策略、失败原因                │
└──────────────────────────────────────────────────────────────┘
          ↓ 推理层熔断后
┌─ 第三层：规划层自修正 ───────────────────────────────────────┐
│ 熔断日志回传，重新评估任务规划                               │
│ 分析失败模式：是工具能力不足？信息缺失？策略错误？             │
│ 生成修正计划或建议用户介入                                   │
│ 修正后重新进入执行，但修正次数上限为2次                       │
│ 2次修正仍失败 → 两次纠正规则触发，建议新会话                 │
└──────────────────────────────────────────────────────────────┘
```

> 引用依据：相关参考 明确"三层防死循环机制必须内置：工具层硬隔离（超时+最大重试）→推理层熔断（连续三次无效推理则中断）→规划层自修正（失败日志回传Lead Agent重新规划）"；相关参考 强调"没有熔断机制的Loop就是烧钱机器"。

### Task 工具子代理使用规范

**核心原则**：子代理是隔离运行的专家，独立上下文窗口，只做一件事，只返回最终结果。

**使用规范**：

1. **独立上下文**：子代理拥有独立的上下文窗口，不共享父级对话历史。所需信息通过任务描述和参数传入。
2. **只返回最终结果**：子代理的探索过程、中间推理、工具调用细节不返回给父级。父级只接收最终的结论或产出。
3. **描述中明确声明**：在 Task 工具调用时，子代理描述中必须包含"只返回最终结果，不返回中间过程"的声明。
4. **三条铁律**：
   - 子代理之间不能互相通信
   - 子代理不能创建新的子代理
   - 所有信息流必须经过父级流转
5. **适用场景**：完全独立的原子任务——安全审查、性能分析、代码风格检查等互不依赖的任务。

**Task 工具调用模板**：

```
使用 Task 工具派发子代理时：
- 明确任务目标和验收标准
- 提供必要的上下文（文件路径、相关规范）
- 在描述中声明"只返回最终结果，不返回中间过程"
- 设定合理的超时时间
```

> 引用依据：相关参考 详解 Sub-Agent 三条铁律和"只返回最终结果"原则；相关参考 源码中子 Agent 使用 minimal Prompt 是务实取舍——不需要完整技能目录和记忆系统，精简 Prompt 把更多窗口留给任务。

## 关键坑点

- **VERIFY 模式在原会话中执行**：这是最常见的违规操作。同一会话中执行+验证违反 Maker/Checker 分离原则，自我评估偏乐观。必须新会话发起。
- **agent_tasks.json 并发写入冲突**：多个会话同时写入可能导致数据丢失。必须使用乐观并发控制，写入前重新读取比对。
- **文件锁粒度过粗**：按目录锁会导致不必要的串行化，按文件锁是最小粒度。不同任务操作同一目录下不同文件应可并行。
- **子代理返回中间过程**：子代理把探索过程全部返回，污染父级上下文。必须在描述中强制声明"只返回最终结果"。
- **死循环未设上限**：Agent 在失败循环中空转烧钱。三层防死循环机制必须全部启用，任何一层都不能省略。
- **任务认领后不释放**：会话异常退出导致任务永远处于 in_progress 状态。30分钟超时自动释放机制必须生效。
- **按"角色"拆分而非"上下文边界"**：Planner→Developer→Tester 的每次交接都产生上下文丢失。正确做法是按"上下文边界"分解任务。

## 可迁移到 AOS 的规则

- AOS Agent 执行模型为单次触发式，状态靠文件系统延续，不依赖运行时内存
- 三种执行模式（PLAN/EXECUTE/VERIFY）严格分离，VERIFY 必须在新会话中发起
- agent_tasks.json 是多会话协同的唯一状态源，所有任务状态变更必须落盘
- 文件级锁 + 乐观并发控制处理多会话并发冲突，30分钟超时自动释放
- 三层防死循环机制全部启用：工具层硬隔离 → 推理层熔断 → 规划层自修正
- Task 工具子代理遵循三条铁律：不互相通信、不创建新子代理、信息流经父级
- 子代理描述中必须声明"只返回最终结果，不返回中间过程"
- 按"上下文边界"分解任务，不按"角色"拆分
- 删除一个 Agent 往往比增加一个 Agent 更值钱——从最简单开始
- 两次纠正规则：同一错误纠正两次失败后，不再继续，建议新会话
