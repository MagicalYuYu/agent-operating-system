## 核心概念

### Harness Engineering 核心原则

Harness Engineering 是 AI 工程的第三阶段演进：Prompt Engineering（让模型听懂）→ Context Engineering（让模型知道）→ Harness Engineering（让模型持续做对）。

**AOS 的 Harness 原则**：每次 Agent 犯错，都工程化一个解决方案，让它永远不再犯同样的错误。修复必须沉淀到环境（AGENTS.md 规则、linter、自动化测试、Skill 经验），而不是留在人脑子里。

**复利效应**：每次犯错 → 环境变强一点 → Agent 下次更少犯错 → 改进速度更快 → Harness 越来越坚固 → Agent 越跑越稳。这个循环不是靠提示词优化或上下文管理能实现的，它需要工程化的结构性修复。

> 引用依据：相关参考 明确 Harness Engineering 核心定义来自 Mitchell Hashimoto（HashiCorp 创始人），"每次 Agent 犯错，都工程化一个解决方案，让它永远不再犯同样的错误"；> 引用依据：相关参考 详解复利效应——"修复必须沉淀到环境里而不是留在人脑子里，这是 Harness 的核心原则"。

## 运行时铁律（强制）

1. **犯错工程化修复流程必须是 Skill 的强制步骤**：不依赖 Agent "记得"去记录错误和修复。修复流程必须嵌入 Skill 步骤序列
2. **记忆更新由用户触发或流程强制**：不假设 Agent 会主动判断"什么值得记"。记忆更新通过用户指令触发（"记住这个经验"）或 Skill 流程中的强制步骤
3. **版本一致性是结构性变更的强制收尾步骤**：每次结构性变更完成后，必须执行全文扫描验证一致性

## 方法论 / 流程

### 第一层：上下文精细化管理

**核心问题**：信息太多（筛选优先级机制）、信息太旧（遗忘和更新机制）、信息冲突（优先级和可信度排序）。

**AGENTS.md 精简原则**：
- 根 AGENTS.md 只放指针和 gotcha，不放全部内容
- 项目本质定义（开头50字）+ 技术栈指针 + 关键 gotcha
- 细节推到子文件：`00_BOOT/rules/*.md` 按模块拆分的增量规则
- 原因：AGENTS.md 每次会话都完整注入，太长挤占有效空间

**压缩指令块**：
- 在 AGENTS.md 中写入压缩指令块，每次压缩后从磁盘重新加载
- 压缩时始终保留：当前任务目标和验收标准、本会话中修改的文件路径及行号、未解决的错误消息、架构决策及其理由、用户明确给出的约束

**规则分层加载**：
- 常驻层（AGENTS.md）：每次会话加载，只放最关键的指针和坑点
- 按需层（Skills）：只在匹配到相关任务时才加载
- 始终可用层（MCP 工具）：不需要注入上下文，Agent 随时可以调用
- 特定任务层（Task 子代理）：独立上下文窗口和工具集

> 引用依据：相关参考 详解压缩指令块——"压缩时，始终保留当前任务目标和验收标准、本会话中修改的文件路径及行号、未解决的错误消息、架构决策及其理由、用户明确给出的约束"；相关参考 强调"根 CLAUDE.md 只放指针和 gotcha，每次会话都完整注入，太长挤占有效空间"。

### 第二层：工具系统可控调用

**工具安全属性声明**：
- 每个工具必须声明三个安全属性：只读（readOnly）/ 破坏性（destructive）/ 可并行（concurrencySafe）
- 类型系统强制要求，漏了编译不过——每把刀都有刀鞘
- 示例：Read/Grep/Glob 标记 readOnly=true + concurrencySafe=true；Edit/Write 标记 readOnly=false + concurrencySafe=false；Bash 标记 readOnly=false + concurrencySafe=false

**暴露数量控制 15-20 个**：
- 工具数量直接影响模型工具选择准确率，超过 15-20 个时选错概率显著上升
- 按场景过滤：当前任务只暴露相关工具，隐藏无关工具
- 高频基础工具直接注入，长尾工具通过按需加载机制延迟加载
- 工具列表按字母排序，保持缓存稳定性

**权限分层管控**：
- 否决规则：settings.json 里的 deny 规则，命中直接拒绝
- 询问规则：某些工具配了"每次都问我"
- 工具自身权限：根据安全属性判断
- 模式决策：PLAN 模式只允许只读工具，EXECUTE 模式允许全部
- Rules 一票否决：Trae Rules 中配置的规则，任何一个返回 block 则整体拒绝

> 引用依据：相关参考 明确"工具定义标准化——每个工具必须声明只读/破坏性/可并行三个安全属性，类型系统强制要求，漏了编译不过"；相关参考 指出"工具超过15-20个时模型选错概率显著上升"；相关参考 详解工具三层过滤模型和 isConcurrencySafe 标记。

### 第三层：任务执行全局编排

**/spec 实现 PLAN 模式**：
- `/spec` 命令触发 PLAN 模式，权限降为只读
- PLAN 模式下只能使用 Read/Grep/Glob 等只读工具
- 计划写入指定路径，用户审批后切换到 EXECUTE 模式执行
- 每5轮对话系统注入提醒：当前处于 PLAN 模式，请勿执行写操作

**失败恢复**：
- 弹性重试策略，不是简单重试：
  - 上下文溢出：自动压缩，最多3次
  - 超时：先检查 token 使用率，判断是否需要压缩
  - 认证失败：轮转 auth profile
  - 速率限制：profile 轮转或模型降级
- 每种失败类型有独立的恢复策略，不是一刀切重试

**任务编排**：
- 复杂任务通过 /spec 拆解为子任务
- 子任务通过 agent_tasks.json 调度和追踪
- 子代理执行独立子任务，只返回最终结果

> 引用依据：相关参考 详解 Plan Mode 通过 EnterPlanMode/ExitPlanMode 两个工具实现，进入后权限降只读，每5轮系统塞小纸条提醒；> 引用依据：相关参考 详解弹性重试——"上下文溢出自动压缩最多3次、超时先检查token使用率、认证失败轮转auth profile、速率限制profile轮转或模型降级"。

### 第四层：记忆与状态分层管理

**四类记忆**：

| 类型 | 说明 | 存储位置 | 示例 |
|------|------|----------|------|
| 用户偏好 | 编辑器、代码风格等个人设置 | 04_MEMORY/preferences/ | "使用 vim 键绑定"、"缩进2空格" |
| 项目约定 | 目录结构、技术栈、构建命令 | 04_MEMORY/conventions/ | "项目使用 Rust + Tokio"、"测试命令 cargo test" |
| 关键决策 | 为什么选这个方案 | 04_MEMORY/decisions/ | "选择 JWT 而非 Session，因为需要无状态认证" |
| 事实信息 | API endpoint、表名等事实 | 04_MEMORY/facts/ | "数据库连接串在 .env 中"、"API 基地址 https://api.example.com" |

**排除规则**：临时性信息、可推导信息、敏感信息不存入记忆。

**INDEX.md 索引**：
- 04_MEMORY/INDEX.md 记录所有记忆条目摘要和位置
- 召回时先查索引筛选相关条目，再按需读取具体文件
- 避免全量加载记忆导致 token 浪费

**记忆提升三条件**（短期记忆提升为长期记忆必须同时满足）：
1. **分数够高**：语义匹配质量好，不是随机命中
2. **被召回次数够多**：不是偶然命中，而是反复需要
3. **被不同查询召回过**：在多个场景下都有用

第三个条件最精妙——区分"用户反复问同一个问题"（系统没回答好）和"知识在多个场景下都有用"（真正值得长期记住）。

> 引用依据：相关参考 详解记忆提升三条件——"分数够高（语义匹配质量好）、被召回次数够多（不是偶然命中）、被不同查询召回过（在多个场景下都有用）。第三个条件最精妙——区分'用户反复问同一个问题'（系统没回答好）和'知识在多个场景下都有用'（真正值得长期记住）"；相关参考 详解记忆系统四类型和索引+独立文件方案。

### 第五层：独立评估与观测体系

**VERIFY 模式新会话**：
- 任务完成后必须在新会话中发起 VERIFY
- 新会话独立上下文，避免自我评估偏乐观
- 审查报告写入 06_LOGS/verify_YYYYMMDD_HHMMSS_{task_id}.md
- 审查结果更新回 agent_tasks.json

**06_LOGS/ 记录**：
- 所有 Agent 执行过程记录到 06_LOGS/
- 日志只追加不修改
- 包含：每步输入/输出/耗时/token 消耗
- Loop 执行日志：06_LOGS/loop_LP_XXXX_YYYYMMDD.log

**可观测性**：
- 全链路日志记录每步操作
- 上下文占用可视化：类似 /context 的彩色网格+各类别明细
- 每30分钟自动提示上下文占用百分比
- 上下文区域模型：规划区(0-40%) → 纯代码区(40-70%) → 转储区(70-85%) → 退出转储区(85-95%) → 死区(95%+)

> 引用依据：相关参考 指出"模型评估自己输出时倾向给偏高评分，因为缺乏外部基准和自我一致性陷阱"，解决方案是引入独立评估模型和量化指标；相关参考 详解上下文区域模型和 /cost 健康度检查。

### 第六层：约束校验与失败恢复

**两次纠正规则**：
- 同一错误纠正两次失败后，不再继续纠正
- 将失败原因写成 3-5 行简报
- 建议用户开启新会话并将简报贴入
- 原因：第三次纠正只是在浪费 token——Agent 已经陷入一个错误的上下文路径

**三级降级策略**：
1. **完整摘要**：正常压缩，保留关键决策和文件路径
2. **排除超大消息**：压缩后仍超限，排除最大的工具输出消息
3. **纯文本降级**：仍然超限，剥离所有格式，只保留纯文本核心信息

**熔断机制**：
- 三层防死循环：工具层硬隔离（超时+最大重试）→ 推理层熔断（连续三次无效推理则中断）→ 规划层自修正（失败日志回传重新规划）
- Loop 熔断：max_iterations 上限 + error_count 阈值 + 连续3次失败暂停
- 上下文熔断：上下文占用到 60-70% 时主动压缩，不等自动压缩（~92%）

> 引用依据：相关参考 明确"纠正Claude同一种错误不要超过两次，第三次只是浪费token——/clear重来，把失败原因写成3-5行简报贴进新会话"；> 引用依据：相关参考 详解三级降级——"完整摘要→排除超大消息→纯文本降级"；相关参考 详解三层防死循环机制。

### 犯错工程化修复流程

```
┌─ Step 1：记录 feedback ─────────────────────────────────────┐
│ Agent 犯错时，记录：                                         │
│ - 错误现象（做了什么不该做的/没做什么该做的）                 │
│ - 错误上下文（当时在做什么任务、用了什么工具）               │
│ - 纠正方式（用户怎么纠正的、正确做法是什么）                 │
│ 记录位置：06_LOGS/feedback_YYYYMMDD_HHMMSS.md               │
└──────────────────────────────────────────────────────────────┘
          ↓
┌─ Step 2：判断修复方式 ─────────────────────────────────────┐
│ 修复方式分三级：                                             │
│ - 规则级：写入 AGENTS.md 或 00_BOOT/rules/*.md              │
│   （适用：行为规范类错误，如"不要直接 export"）              │
│ - Skill级：更新相关 Skill 的经验/坑点清单                    │
│   （适用：技能执行类错误，如"API调用顺序不对"）              │
│ - 工具级：增加工具约束或安全属性                             │
│   （适用：工具使用类错误，如"误用破坏性工具"）               │
└──────────────────────────────────────────────────────────────┘
          ↓
┌─ Step 3：沉淀到环境 ───────────────────────────────────────┐
│ 将修复写入对应的持久化位置：                                 │
│ - 规则级 → AGENTS.md 或 00_BOOT/rules/                     │
│ - Skill级 → 03_TOOLS/skills/{skill}/SKILL.md               │
│ - 工具级 → 工具定义的安全属性声明                           │
│ 关键：修复必须写入文件系统，不能只在当前会话中口头纠正       │
└──────────────────────────────────────────────────────────────┘
          ↓
┌─ Step 4：更新演化记录 ─────────────────────────────────────┐
│ 在 04_MEMORY/decisions/ 中记录：                            │
│ - 修复时间、错误类型、修复方式                              │
│ - 预期效果：下次遇到同类场景时不再犯错                      │
│ - 验证方式：后续会话中观察是否复发                          │
└──────────────────────────────────────────────────────────────┘
```

> 引用依据：相关参考 核心原则——"绝大多数人遇到Agent犯错是'骂两句，手动改掉，祈祷下次别再犯'。Harness的做法是每次犯错都问：我能不能把这个错误永久性地修到环境里，让它下次在结构上就不可能再犯？"

### Trae 适配说明

AOS 运行在 Trae CODE 模式下，部分 Claude Code 原生机制需要适配替代方案：

| Claude Code 机制 | Trae CODE 适配方案 | 说明 |
|------------------|-------------------|------|
| Hooks（PreToolUse/PostToolUse/Stop） | **Rules**（.trae/rules/） | Trae 用 Rules 替代 Hooks，在规则文件中声明行为约束 |
| Worktree（--worktree 隔离） | **02_SANDBOX/** | AOS 的 02_SANDBOX 目录承担隔离实验职责，所有不确定操作进入沙箱 |
| /compact（上下文压缩） | **压缩指令块** | 在 AGENTS.md 中写入压缩指令块，压缩时从磁盘重新加载 |
| CLAUDE.md | **AGENTS.md** | AOS 使用 AGENTS.md 作为系统规则入口 |
| .claude/rules/*.md | **00_BOOT/rules/*.md** | AOS 的规则分层存储在 00_BOOT 目录下 |
| Skills | **03_TOOLS/skills/** | AOS 的 Skill 存储在 03_TOOLS 目录下 |
| Memory | **04_MEMORY/** | AOS 的记忆系统存储在 04_MEMORY 目录下 |
| Subagents | **Task 工具子代理** | Trae 通过 Task 工具实现子代理，独立上下文，只返回最终结果 |
| /loop | **Scheduled Loop** | AOS 通过 Trae Schedule 工具实现定时 Loop |

> 引用依据：相关参考 详解 Claude Code 五层 Harness 配置（CLAUDE.md → Hooks → Skills → LSP/MCP → Subagents），AOS 需要适配 Trae CODE 模式的对应机制；相关参考 详解 /compact 压缩指令块替代方案。

## 关键坑点

- **AGENTS.md 写太长**：每次会话完整注入，太长挤占有效空间。根文件只放指针和 gotcha，细节推到 00_BOOT/rules/*.md 子文件。
- **工具暴露过多**：超过 15-20 个时模型选错概率显著上升。必须按场景过滤，高频基础工具直接注入，长尾工具按需加载。
- **自我评估偏乐观**：模型评估自己输出时倾向给偏高评分。VERIFY 必须在新会话中发起，引入独立评估。
- **两次纠正后继续纠正**：第三次纠正只是在浪费 token。必须执行两次纠正规则，建议新会话。
- **修复只留在当前会话**：口头纠正不等于工程化修复。修复必须沉淀到环境（AGENTS.md/Skill/工具属性），否则下次会话还会犯同样的错。
- **记忆膨胀**：所有记忆全塞一个文件，每次全读一遍浪费 token。必须用 INDEX.md 索引 + 独立文件方案按需加载。
- **压缩不带 hint**：不带指令的压缩产生通用摘要，丢失关键信息。压缩指令块必须写入 AGENTS.md。
- **Rules 不等于 Hooks**：Trae Rules 是声明式规则约束，不是 Claude Code Hooks 的命令式触发器。不能直接迁移 Hook 脚本，需要重新设计规则逻辑。

## 可迁移到 AOS 的规则

- 采用 Harness 六层架构作为 Agent 可靠性保障框架
- 每次 Agent 犯错必须工程化结构性修复，修复沉淀到环境而非人脑
- AGENTS.md 精简原则：只放指针和 gotcha，细节推到 00_BOOT/rules/*.md
- 压缩指令块写入 AGENTS.md，确保关键信息在压缩后不丢失
- 工具定义强制声明三个安全属性（只读/破坏性/可并行），类型系统强制
- 工具按场景过滤，暴露工具数控制在 15-20 个以内
- /spec 实现 PLAN 模式，权限降只读，每5轮注入提醒
- 失败恢复采用弹性重试策略，每种失败类型有独立恢复策略
- 记忆提升三条件：分数高+召回多+不同查询召回（区分系统问题和真正有用的知识）
- 独立评估体系：VERIFY 必须在新会话中发起，不依赖模型自评
- 两次纠正规则：同一错误纠正两次失败后建议新会话
- 失败恢复采用三级降级策略：完整摘要 → 排除超大消息 → 纯文本降级
- Trae 适配：Hook 用 Rules 替代、Worktree 用 02_SANDBOX 替代、/compact 用压缩指令块替代
- 规则文件定期清理去重，避免膨胀导致模型注意力分散
- 优先使用成熟技术而非最新框架，降低模型出错率
