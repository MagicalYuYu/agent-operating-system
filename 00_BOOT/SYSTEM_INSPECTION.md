# AOS 系统级巡检流程规范

> **版本**：1.0.0
> **触发层级**：Layer 3（用户显式触发，手动开启）
> **执行方式**：用户每日首次开工前，单独发起一个专用会话，指令 Agent 执行本规范
> **文件性质**：AOS 官方规范文档（系统规则层）

---

## 0. 文档定位

本规范是 **系统级巡检的 Agent 执行指令**。用户每日首次开工前，**单独发起一个专用会话**，将本规范交给 Agent，Agent 按本规范完成系统级巡检后，会话结束。

**与项目级巡检的关系**：
- 系统级巡检在**独立专用会话**中执行，不与任何项目工作混合
- 项目级巡检是**项目工作会话的前置环节**，见 `00_BOOT/PROJECT_INSPECTION.md`
- 两者完全分离，本文件不涉及项目级巡检内容

**核心目标**：
- 保持 AOS 全局文件系统健康，防止冗余堆积
- 确保全局状态文件与实际一致，避免"空洞引用"
- 提供清晰的文件处置决策框架
- 通过固定流程降低遗漏风险

**验证工具说明**：
本规范涉及的所有文件存在性检查必须使用可靠工具验证。注意：部分 IDE 内置的 LS 工具可能因缓存或权限问题返回错误结果，**必须使用 PowerShell `Test-Path` 或 `Get-ChildItem -Force` 进行最终验证**，避免误报。

## 1. 巡检原则

| 原则 | 说明 |
|------|------|
| **先诊断后处置** | 所有文件删除/迁移操作必须先出具报告，经用户确认后方可执行 |
| **状态先行** | 优先检查状态文件（SYSTEM_STATE/INDEX）一致性，状态错乱则后续工作无意义 |
| **分类清晰** | 每个文件必须归入明确类别，禁止"暂存"模糊状态 |
| **可追溯** | 所有处置操作记录到 06_LOGS/，删除操作需备份到 99_ARCHIVE/ |
| **最小破坏** | 优先整合/迁移，次选归档，最后才删除 |
| **路径层级规范** | 所有文件存储路径必须按"项目→模块→类型→日期"分层，禁止扁平混存 |

## 2. 文件路径层级规范（强制）

> **背景**：AOS 多个目录（尤其 06_LOGS/、07_EXPORTS/、08_INBOX/）此前的分层规则过浅，导致同一目录下混杂不同项目、不同模块、不同类型的文件，后续管理混乱。本节定义统一的路径层级规范。

### 2.1 通用层级模型

```
{AOS目录}/{项目名}/{模块名}/{类型}/{YYYYMMDD}_{描述}.{ext}
```

| 层级 | 含义 | 示例 |
|------|------|------|
| 第1层 | AOS 顶级目录 | 06_LOGS/、07_EXPORTS/、08_INBOX/ |
| 第2层 | 项目名（或 aos_system 表示系统级） | example_bot/、example_web/、aos_system/ |
| 第3层 | 模块名（按功能划分） | engine/、platform/、renderer/、inspection/ |
| 第4层 | 类型（按日志/文件类型细分，按需） | runtime/、error/、access/、review/ |
| 文件名 | 日期+描述 | 20260625_inspection_review.md |

### 2.3 层级适配原则

| 原则 | 说明 |
|------|------|
| **按需下沉** | 层级不是越深越好，简单项目可止于第2-3层，复杂项目才下沉到第4层 |
| **模块名稳定** | 模块名应与项目实际架构对应，不随重构频繁变动 |
| **类型可扩展** | 类型层按需新增，但同一模块下的类型必须互斥 |
| **系统级独立** | 非项目专属的文件使用 `aos_system/` 作为第2层 |
| **日期格式统一** | 文件名中的日期一律使用 YYYYMMDD 格式 |

## 3. 系统级巡检流程

系统级巡检覆盖以下阶段（仅全局项，不深入项目内部）：

```
┌─────────────────────────────────────────────────────────────┐
│                  系统级巡检流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  阶段一：系统状态自检                                         │
│    ├─ 1.1 读取 SYSTEM_STATE.md                              │
│    ├─ 1.2 读取 SKILL_REGISTRY.md                            │
│    ├─ 1.3 读取 04_MEMORY/INDEX.md                           │
│    └─ 1.4 读取 agent_tasks.json（如存在）                    │
│                                                             │
│  阶段二：项目识别与状态核对（仅扫描目录，不深入）             │
│    ├─ 2.1 扫描 01_PROJECTS/ 实际目录                        │
│    └─ 2.2 对比项目记忆（04_MEMORY/project/）与实际目录       │
│                                                             │
│  阶段三：文件结构健康检查（仅全局项）                         │
│    ├─ 3.1 空洞引用扫描（INDEX/_index/SKILL_REGISTRY）        │
│    ├─ 3.2 冗余文件扫描（构建产物/临时文件/重复文件）          │
│    ├─ 3.3 路径合规检查（按目录强约束表）                      │
│    ├─ 3.4 路径层级检查（按第2节层级规范）                    │
│    └─ 3.5 08_INBOX 清理评估                                 │
│                                                             │
│  阶段四：文件分类与处置建议（仅全局项）                       │
│    ├─ 4.1 生成分类报告（四类：冗余/过时/留存/迁移）          │
│    ├─ 4.2 弹窗展示报告，等待用户确认                         │
│    └─ 4.3 执行处置（删除/迁移/整合/归档）                    │
│                                                             │
│  阶段五：状态更新与收尾                                       │
│    ├─ 5.1 更新 SYSTEM_STATE.md（自检记录 + 事件日志）        │
│    ├─ 5.2 更新 04_MEMORY/INDEX.md（如有变更）                │
```

**系统级巡检不覆盖**（由项目级巡检负责）：
- 项目内文件深入检查
- 项目 STATUS.md 详细核对
- 双机同步检查

## 4. Agent 执行指令（分阶段）

### 阶段一：系统状态自检

Agent 必须依次执行以下检查，并记录结果：

| 步骤 | 检查项 | 数据源 | 异常处理 |
|------|--------|--------|----------|
| 1.1 | 系统版本一致性 | 00_BOOT/SYSTEM_STATE.md vs AGENTS.md | 不一致则提示用户 |
| 1.2 | 组件状态 | 00_BOOT/SYSTEM_STATE.md | READY 状态需确认是否需激活 |
| 1.3 | Harness Framework | 00_BOOT/SYSTEM_STATE.md + AGENTS.md | 状态异常则提示用户检查 |
| 1.4 | Skill 注册表 | 00_BOOT/SKILL_REGISTRY.md | 路径无效则标记并提示 |
| 1.5 | 记忆索引 | 04_MEMORY/INDEX.md | 超 200 行/25KB 触发截断 |
| 1.6 | 待办任务 | 04_MEMORY/agent_pool/agent_tasks.json | 有待办则提示用户优先处理 |

### 阶段二：项目识别与状态核对（仅扫描，不深入）

| 步骤 | 检查项 | 工具 | 异常处理 |
|------|--------|------|----------|
| 2.1 | 项目目录扫描 | `Get-ChildItem -Force d:\AOS\01_PROJECTS` | 空目录或仅有 _example_game_localization 则提示 |
| 2.2 | 项目记忆一致性 | 对比 04_MEMORY/project/proj_*.md 与实际目录 | 记忆引用了不存在的项目 → 标记为"过时记忆" |
| 2.3 | 项目完整性 | 每个项目必须含 AGENTS.md/README.md/STATUS.md | 缺失则标记为"未完成初始化" |
| 2.4 | 双机项目区分 | laptop 不存在但服务器存在的项目不算"缺失" | 参考 AGENTS.md 双机环境规则 |

### 阶段三：文件结构健康检查（仅全局项）

#### 3.1 空洞引用扫描

扫描以下索引文件，验证每条引用是否指向真实存在的文件（使用 `Test-Path` 验证）：

| 索引文件 | 扫描范围 | 处置 |
|----------|----------|------|
| 04_MEMORY/INDEX.md | user/feedback/reference/project 下的引用 | 引用失效 → 标记为"过时需整合" |
| 09_REFERENCE/_index.md | system/web 下的引用 | 引用失效 → 从索引移除或补建文件 |
| 00_BOOT/SKILL_REGISTRY.md | 03_TOOLS/skills/ 下的路径 | 路径无效 → 标记 Skill 为 INACTIVE |
| 00_BOOT/SYSTEM_STATE.md | 事件日志中的项目引用 | 项目不存在 → 标记为"状态不一致" |

#### 3.2 冗余文件扫描

| 扫描目标 | 判定标准 | 处置建议 |
|----------|----------|----------|
| PyInstaller 构建产物 | `03_TOOLS/*/build/` 目录 | 删除（可随时重新生成） |
| PyInstaller 打包产物 | `03_TOOLS/*/dist/` 目录 | 迁移至 07_EXPORTS/{project}/ 或删除 |
| 临时缓存 | `05_CACHE/` 下非 README 文件 | 删除 |
| 重复文件 | 同名文件出现在多个目录 | 保留规范路径下的，删除其他 |
| 旧版备份 | 文件名含 _old/_backup/_v1 等后缀 | 归档至 99_ARCHIVE/ 或删除 |
| 空目录 | 无任何文件的目录（除 README.md） | 删除或补充 README.md |

#### 3.3 路径合规检查

按 AGENTS.md 目录强约束表验证文件存放：

| 目录 | 允许内容 | 违规示例 |
|------|----------|----------|
| 00_BOOT/ | 仅系统规则文件 | 出现项目文档 |
| 01_PROJECTS/ | 仅项目子目录 | 出现散落脚本 |
| 03_TOOLS/ | Skill/Loop/Agent/脚本模块/adapters 适配器 | 出现项目数据 |
| 03_TOOLS/adapters/ | 各 IDE/Agent 适配规则（如 claude_code/rules/） | 出现项目专属配置 |
| 04_MEMORY/ | 仅状态持久化文件 | 出现代码文件 |
| 07_EXPORTS/ | 仅导出产物 | 出现源代码 |
| 08_INBOX/raw/ | 按日期分层的外部输入原始文件 | 出现处理后的知识库文件 |
| 08_INBOX/migration/ | 迁移状态文件（迁移完成后归档至99_ARCHIVE/） | 出现非迁移相关内容 |
| 09_REFERENCE/ | 仅参考知识 | 出现项目专属内容 |
| 99_ARCHIVE/ | 仅历史归档（只读） | 出现活跃项目文件 |

#### 3.4 路径层级检查

按本规范第2节层级规范检查文件存储路径：

| 检查项 | 判定标准 | 处置 |
|--------|----------|------|
| 层级深度 | 06_LOGS/ 下文件至少在第2层（项目名）下 | 扁平存放的文件标记为 D类（迁移） |
| 模块名合理性 | 模块名应与项目架构对应 | 不合理模块名提示用户重命名 |
| 类型互斥性 | 同一模块下类型目录不重叠 | 重叠类型合并 |
| 系统级识别 | 非项目专属文件应在 aos_system/ 下 | 误放入项目的系统文件迁移至 aos_system/ |

#### 3.5 08_INBOX 清理评估

| 检查项 | 判定标准 | 处置建议 |
|--------|----------|----------|
| raw/ 文件时效 | 超过 30 天且已入库 09_REFERENCE/ 的原始文件 | 归档至 99_ARCHIVE/ 或删除（已入库可删） |
| raw/ 未入库文件 | 超过 7 天未处理的外部输入 | 提示用户处理或归档 |
| raw/ 层级合规 | 文件应按日期分层（raw/YYYYMMDD/） | 不合规则迁移至日期子目录 |
| migration/ 状态文件 | 迁移已完成且无后续操作的状态文件 | 归档至 99_ARCHIVE/migration/ |
| migration/ chat_extracts/ | 已完成迁移的对话提取内容 | 归档或删除（含隐私内容，删除前需脱敏） |

### 阶段四：文件分类与处置（仅全局项）

#### 4.1 四类分类标准

**A 类：完全无用的冗余文件**

| 判定条件 | 示例 | 处置 |
|----------|------|------|
| 可随时重新生成的构建产物 | PyInstaller build/ 目录 | 直接删除 |
| 纯临时数据 | 05_CACHE/ 下非重要文件 | 直接删除 |
| 重复存在的副本 | 多处复制的同一文件 | 保留规范路径下的，删除其他 |
| 空目录占位 | 无内容且无 README 的目录 | 删除 |

**B 类：内容过时需整合至其他文件的文件**

| 判定条件 | 示例 | 处置 |
|----------|------|------|
| 索引引用了不存在的目标 | INDEX.md 引用缺失文件 | 修复索引或补建文件 |
| 状态与实际不符 | SYSTEM_STATE.md 记录了不存在的项目 | 更新状态文件 |
| 旧版本残留 | 已被新版本替代但未清理 | 归档至 99_ARCHIVE/ 后删除 |
| 内容已被迁移但原文件未删 | 迁移后源文件残留 | 确认迁移完成后删除 |

**C 类：需留存的文件**

| 判定条件 | 示例 | 处置 |
|----------|------|------|
| 系统规则文件 | 00_BOOT/ 下所有文件 | 保留原位 |
| 活跃项目文件 | 01_PROJECTS/{active}/ 下文件 | 保留原位 |
| 已注册 Skill | 03_TOOLS/skills/ 下 ACTIVE Skill | 保留原位 |
| 参考知识库 | 09_REFERENCE/ 下文档 | 保留原位 |
| 记忆文件 | 04_MEMORY/ 下有效记忆 | 保留原位 |
| 凭据文件 | 04_MEMORY/credentials.json | 保留原位（敏感） |

**D 类：存储路径不合理需迁移的文件**

| 判定条件 | 示例 | 处置 |
|----------|------|------|
| 违反目录强约束 | 项目数据出现在 03_TOOLS/ | 迁移至 01_PROJECTS/ |
| 打包产物在源码目录 | dist/ 在 03_TOOLS/ 下 | 迁移至 07_EXPORTS/ |
| 构建中间产物在工作目录 | build/ 在源码旁 | 迁移至 05_CACHE/ 或删除 |
| 日志在非日志目录 | .log 文件散落各处 | 迁移至 06_LOGS/{project}/ |
| 路径层级不合规 | 06_LOGS/ 下扁平存放 | 迁移至 06_LOGS/{project}/{module}/ |

#### 4.2 处置决策矩阵

```
文件存在？
├─ 否 → 检查是否被索引引用
│       ├─ 是 → B类（修复索引或补建）
│       └─ 否 → 无需处理
└─ 是 → 检查路径合规性
        ├─ 不合规 → D类（迁移至规范路径）
        └─ 合规 → 检查内容有效性
                ├─ 可重新生成 → A类（删除）
                ├─ 内容过时 → B类（整合/归档）
                └─ 内容有效 → C类（保留）
```

### 阶段五：状态更新与收尾

| 步骤 | 更新目标 | 更新内容 | 触发条件 |
|------|----------|----------|----------|
| 5.1 | 00_BOOT/SYSTEM_STATE.md | 自检记录 + 事件日志 | 每次巡检必更新 |
| 5.2 | 04_MEMORY/INDEX.md | 记忆索引条目 | 记忆有变更时更新 |
| 5.3 | 06_LOGS/aos_system/inspection/{YYYYMMDD}_system.md | 系统级巡检日志 | 每次巡检必写入 |

## 5. 巡检报告输出格式

Agent 完成阶段一至三后，必须生成巡检报告并弹窗展示给用户。报告格式如下：

```markdown
# AOS 系统级巡检报告 — {YYYY-MM-DD}

| 项目 | 值 |
|------|-----|
| 巡检时间 | {YYYY-MM-DD HH:MM} |
| AOS 版本 | {version} |
| 巡检类型 | 系统级 |

## 系统状态
| 检查项 | 结果 | 备注 |
|--------|------|------|
| 版本一致性 | PASS/FAIL | ... |
| Skill 注册表 | PASS/FAIL | ... |
| 记忆索引 | PASS/FAIL | ... |

## 文件分类结果

### A 类：完全无用的冗余文件（建议删除）
| 文件路径 | 类型 | 大小 | 删除原因 |
|----------|------|------|----------|

### B 类：内容过时需整合（建议修复/归档）
| 文件路径 | 问题 | 建议处置 |
|----------|------|----------|

### C 类：需留存的文件（无需操作）
```

## 6. 异常处理

| 异常场景 | 处理方式 |
|----------|----------|
| 发现关键状态文件损坏 | 立即停止巡检，弹窗警告用户，等待人工干预 |
| 用户拒绝处置建议 | 保留文件原状，在日志中记录"用户拒绝"，继续后续巡检 |
| 删除操作误删重要文件 | 立即停止，从 99_ARCHIVE/ 恢复（如已归档），报告用户 |
| LS 工具返回异常结果 | 改用 `Test-Path` / `Get-ChildItem -Force` 验证，不依赖 LS 工具结论 |

## 7. 巡检频率与触发机制

| 触发方式 | 频率 | 说明 |
|----------|------|------|
| 用户显式触发 | 每日1次 | 用户每日首次开工前，单独发起专用会话执行 |
| Agent 主动建议 | 检测到状态异常时 | Agent 发现状态不一致时建议用户发起巡检 |

**重要说明**：由于 AOS 是单次对话触发式 Agent，不存在后台常驻进程，因此"每日自动巡检"在技术上是不可行的。巡检必须由用户显式触发。本规范属于 Layer 3（用户指令显式触发），不属于 Layer 1（自动触发）。

## 8. 巡检日志路径规范

系统级巡检日志存储路径：

| 路径 | 说明 |
|------|------|
| `06_LOGS/aos_system/inspection/{YYYYMMDD}_system.md` | 每日1个文件 |
