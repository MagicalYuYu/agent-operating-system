# AOS 项目级巡检流程规范

> **版本**：1.0.0
> **触发层级**：Layer 3（用户显式触发，手动开启）
> **执行方式**：用户发起项目工作会话时，Agent 先执行本规范完成项目级巡检，再开始当日项目工作
> **文件性质**：AOS 官方规范文档（系统规则层）

---

## 0. 文档定位

本规范是 **项目级巡检的 Agent 执行指令**。用户发起项目工作会话时，Agent **先执行项目级巡检**，完成后再开始当日项目工作。Agent 熟悉当前项目，可准确判断文件健康状态。

**与系统级巡检的关系**：
- 项目级巡检是**项目工作会话的前置环节**
- 系统级巡检在**独立专用会话**中执行，见 `00_BOOT/SYSTEM_INSPECTION.md`
- 两者完全分离，本文件不涉及系统级巡检内容

**核心目标**：
- 保持当前项目文件健康，防止冗余堆积
- 确保项目状态文件（STATUS.md/AGENTS.md）与实际一致
- 核对双机同步状态（如涉及服务器项目）
- 通过固定流程降低遗漏风险

**验证工具说明**：
本规范涉及的所有文件存在性检查必须使用可靠工具验证。注意：部分 IDE 内置的 LS 工具可能因缓存或权限问题返回错误结果，**必须使用 PowerShell `Test-Path` 或 `Get-ChildItem -Force` 进行最终验证**，避免误报。

## 1. 前置检查（关键）

Agent 执行项目级巡检前，**必须先检查系统级巡检是否已完成**：

| 检查项 | 方式 | 结果处理 |
|--------|------|----------|
| 系统级巡检日志是否存在 | `Test-Path 06_LOGS/aos_system/inspection/{YYYYMMDD}_system.md` | 存在 → 继续执行项目级巡检；不存在 → 提示用户"今日尚未执行系统级巡检，建议先单独发起系统级巡检会话" |

**用户选择权**：即使用户选择继续，Agent 也应执行项目级巡检，但在日志中标注"系统级巡检未完成"。

---

## 2. 巡检原则

| 原则 | 说明 |
|------|------|
| **先诊断后处置** | 所有文件删除/迁移操作必须先出具报告，经用户确认后方可执行 |
| **状态先行** | 优先检查项目状态文件（STATUS.md/AGENTS.md）一致性 |
| **分类清晰** | 每个文件必须归入明确类别，禁止"暂存"模糊状态 |
| **可追溯** | 所有处置操作记录到 06_LOGS/，删除操作需备份到 99_ARCHIVE/ |
| **最小破坏** | 优先整合/迁移，次选归档，最后才删除 |
| **双机意识** | 涉及服务器项目时，必须区分 laptop/服务器路径，不可混淆 |
| **路径层级规范** | 所有文件存储路径必须按"项目→模块→类型→日期"分层，禁止扁平混存 |
| **仅当前项目** | 巡检范围仅限当前项目目录，不涉及其他项目 |

## 3. 文件路径层级规范（强制）

> **背景**：AOS 多个目录（尤其 06_LOGS/、07_EXPORTS/、08_INBOX/）此前的分层规则过浅，导致同一目录下混杂不同项目、不同模块、不同类型的文件，后续管理混乱。本节定义统一的路径层级规范。

### 3.1 通用层级模型

```
{AOS目录}/{项目名}/{模块名}/{类型}/{YYYYMMDD}_{描述}.{ext}
```

| 层级 | 含义 | 示例 |
|------|------|------|
| 第1层 | AOS 顶级目录 | 06_LOGS/、07_EXPORTS/、08_INBOX/ |
| 第2层 | 项目名（或 aos_system 表示系统级） | astrbot/、bishoujo-web/、aos_system/ |
| 第3层 | 模块名（按功能划分） | engine/、platform/、renderer/、inspection/ |
| 第4层 | 类型（按日志/文件类型细分，按需） | runtime/、error/、access/、review/ |
| 文件名 | 日期+描述 | 20260625_inspection_project.md |

### 3.2 项目级巡检日志路径规范

项目级巡检日志存储路径：

| 路径 | 说明 |
|------|------|
| `06_LOGS/{project_name}/inspection/{YYYYMMDD}_project.md` | 每项目每日1个文件 |

Agent 通过检查该日志是否存在，判断是否需要执行项目级巡检。若已存在则跳过，直接开始项目工作。

### 3.3 层级适配原则

| 原则 | 说明 |
|------|------|
| **按需下沉** | 层级不是越深越好，简单项目可止于第2-3层，复杂项目才下沉到第4层 |
| **模块名稳定** | 模块名应与项目实际架构对应，不随重构频繁变动 |
| **类型可扩展** | 类型层按需新增，但同一模块下的类型必须互斥 |
| **系统级独立** | 非项目专属的文件使用 `aos_system/` 作为第2层 |
| **日期格式统一** | 文件名中的日期一律使用 YYYYMMDD 格式 |

## 4. 项目级巡检流程

项目级巡检仅覆盖当前项目，不涉及其他项目：

```
┌─────────────────────────────────────────────────────────────┐
│                  项目级巡检流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  前置检查：系统级巡检日志是否存在                             │
│    ├─ 不存在 → 提示用户（用户可选择继续或先执行系统级巡检）   │
│    └─ 已存在 → 继续                                         │
│                                                             │
│  阶段二：当前项目状态核对                                     │
│    ├─ 2.1 读取项目 AGENTS.md（加载项目专属配置）             │
│    ├─ 2.2 读取项目 STATUS.md（确认当前阶段）                 │
│    ├─ 2.3 读取项目记忆 proj_{name}.md（核对动态事实）        │
│    └─ 2.4 项目状态停滞检查（超过7天则提示）                  │
│                                                             │
│  阶段三：当前项目文件健康检查                                 │
│    ├─ 3.1 项目内空洞引用扫描（reference_links.md等）         │
│    ├─ 3.2 项目内冗余文件扫描（backup/临时文件等）            │
│    ├─ 3.3 项目内路径合规检查                                 │
│    ├─ 3.4 项目内路径层级检查                                 │
│    └─ 3.5 双机同步检查（如涉及服务器项目）                   │
│                                                             │
│  阶段四：文件分类与处置建议（仅当前项目文件）                 │
│    ├─ 4.1 生成分类报告（四类：冗余/过时/留存/迁移）          │
│    ├─ 4.2 弹窗展示报告，等待用户确认                         │
│    └─ 4.3 执行处置（删除/迁移/整合/归档）                    │
│                                                             │
│  阶段五：状态更新与收尾                                       │
│    ├─ 5.1 更新项目 STATUS.md                                 │
│    ├─ 5.2 更新项目记忆 proj_{name}.md                        │
│    └─ 5.3 写入项目级巡检日志                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**项目级巡检不覆盖**（由系统级巡检负责）：
- 系统状态自检（SYSTEM_STATE.md/SKILL_REGISTRY.md/INDEX.md）
- 全局空洞引用扫描
- 全局冗余文件扫描
- 全局路径合规检查

## 5. Agent 执行指令（分阶段）

### 前置检查

| 步骤 | 检查项 | 方式 | 异常处理 |
|------|--------|------|----------|
| 0.1 | 系统级巡检日志 | `Test-Path 06_LOGS/aos_system/inspection/{YYYYMMDD}_system.md` | 不存在则提示用户，用户可选择继续或先执行系统级巡检 |
| 0.2 | 项目级巡检日志 | `Test-Path 06_LOGS/{project_name}/inspection/{YYYYMMDD}_project.md` | 已存在则跳过巡检，直接开始项目工作 |

### 阶段二：当前项目状态核对

| 步骤 | 检查项 | 数据源 | 异常处理 |
|------|--------|--------|----------|
| 2.1 | 项目配置加载 | `01_PROJECTS/{name}/AGENTS.md` | 缺失则提示"项目未完成初始化" |
| 2.2 | 项目状态 | `01_PROJECTS/{name}/STATUS.md` | 缺失则提示；阶段停滞超过7天则提示 |
| 2.3 | 项目记忆核对 | `04_MEMORY/project/proj_{name}.md` | 记忆与实际不符则标记为"过时记忆" |
| 2.4 | 项目完整性 | AGENTS.md/README.md/STATUS.md 三件齐全 | 缺失则标记为"未完成初始化" |

### 阶段三：当前项目文件健康检查

#### 3.1 项目内空洞引用扫描

| 扫描目标 | 检查内容 | 处置 |
|----------|----------|------|
| reference_links.md | 验证每条知识引用路径是否存在 | 引用失效 → 标记为"过时需整合" |
| AGENTS.md 相关项目 | 验证相关项目路径是否存在 | 路径失效 → 标记为"状态不一致" |
| STATUS.md 版本引用 | 验证引用的版本/备份是否存在 | 失效 → 标记为"过时" |

#### 3.2 项目内冗余文件扫描

| 扫描目标 | 判定标准 | 处置建议 |
|----------|----------|----------|
| backup/ 目录 | 超过30天的备份文件 | 归档至 99_ARCHIVE/ 或删除 |
| 临时文件 | 项目目录下的 .tmp/.bak/.log 文件 | 删除或迁移至 06_LOGS/ |
| 旧版备份 | 文件名含 _old/_backup/_v1 等后缀 | 归档至 99_ARCHIVE/ 或删除 |
| 空目录 | 无任何文件的目录（除 README.md） | 删除或补充 README.md |
| 构建产物 | 项目内的 build/dist 目录（如有） | 迁移至 07_EXPORTS/ 或删除 |

#### 3.3 项目内路径合规检查

按 AGENTS.md 项目内标准结构验证：

| 检查项 | 判定标准 | 处置 |
|--------|----------|------|
| 必需文件 | AGENTS.md/README.md/STATUS.md 三件齐全 | 缺失则补建 |
| src/ 结构 | 单一项目直接存放代码，项目集存放子项目目录 | 违规则提示重构 |
| docs/ 用途 | 存放项目文档（讨论记录、开发规划、设计文档） | 违规则迁移 |
| config/ 用途 | 存放项目专属配置 | 违规则迁移 |
| 禁止嵌套 | src/ 下不再嵌套与项目同名的目录 | 违规则提示重构 |

#### 3.4 项目内路径层级检查

按本规范第3节层级规范检查项目相关文件存储路径：

| 检查项 | 判定标准 | 处置 |
|--------|----------|------|
| 项目日志层级 | 06_LOGS/{project_name}/ 下至少有模块名层 | 扁平存放的文件标记为 D类（迁移） |
| 项目导出层级 | 07_EXPORTS/{project_name}/ 下按类型分层 | 扁平存放的文件标记为 D类（迁移） |
| 模块名合理性 | 模块名应与项目架构对应 | 不合理模块名提示用户重命名 |

#### 3.5 双机同步检查（如涉及服务器项目）

| 检查项 | 方式 | 异常处理 |
|--------|------|----------|
| 服务器可达性 | `Test-Path \\SERVER_NAME\c\AOS\` | 不可达则提示用户检查服务器 |
| 项目目录一致性 | 对比 laptop 与服务器 01_PROJECTS/{name}/ | 不一致则提示同步 |
| 运行时数据保护 | 确认服务器 data/ 目录未被本地覆盖 | 发现风险立即警告 |
| 版本同步状态 | 对比关键文件版本（如 CHANGELOG.md） | 不一致则提示同步 |

### 阶段四：文件分类与处置（仅当前项目文件）

#### 4.1 四类分类标准

**A 类：完全无用的冗余文件**

| 判定条件 | 示例 | 处置 |
|----------|------|------|
| 可随时重新生成的构建产物 | 项目内 build/ 目录 | 直接删除 |
| 纯临时数据 | .tmp/.bak 文件 | 直接删除 |
| 重复存在的副本 | 多处复制的同一文件 | 保留规范路径下的，删除其他 |
| 空目录占位 | 无内容且无 README 的目录 | 删除 |

**B 类：内容过时需整合至其他文件的文件**

| 判定条件 | 示例 | 处置 |
|----------|------|------|
| 索引引用了不存在的目标 | reference_links.md 引用缺失文件 | 修复索引或补建文件 |
| 状态与实际不符 | STATUS.md 记录了不存在的版本 | 更新状态文件 |
| 旧版本残留 | 已被新版本替代但未清理 | 归档至 99_ARCHIVE/ 后删除 |
| 内容已被迁移但原文件未删 | 迁移后源文件残留 | 确认迁移完成后删除 |

**C 类：需留存的文件**

| 判定条件 | 示例 | 处置 |
|----------|------|------|
| 项目配置文件 | AGENTS.md/README.md/STATUS.md | 保留原位 |
| 活跃源代码 | src/ 下当前版本代码 | 保留原位 |
| 项目文档 | docs/ 下文档 | 保留原位 |
| 项目配置 | config/ 下配置 | 保留原位 |
| 当前版本备份 | backup/ 下当前版本备份 | 保留原位 |

**D 类：存储路径不合理需迁移的文件**

| 判定条件 | 示例 | 处置 |
|----------|------|------|
| 违反项目结构 | 项目数据出现在 docs/ | 迁移至 src/ 或 config/ |
| 日志在非日志目录 | .log 文件在项目根目录 | 迁移至 06_LOGS/{project}/ |
| 路径层级不合规 | 06_LOGS/ 下扁平存放 | 迁移至 06_LOGS/{project}/{module}/ |
| 导出产物在源码目录 | dist/ 在 src/ 下 | 迁移至 07_EXPORTS/{project}/ |

#### 4.2 处置决策矩阵

```
文件存在？
├─ 否 → 检查是否被索引引用
│       ├─ 是 → B类（修复索引或补建）
│       └─ 否 → 无需处理
└─ 是 → 检查路径合规性
        ├─ 不合规 → D类（迁移至规范路径）
        └─ 合规 → 检查内容有效性
                ├─ 可重新生成 → A类（删除）
                ├─ 内容过时 → B类（整合/归档）
                └─ 内容有效 → C类（保留）
```

### 阶段五：状态更新与收尾

| 步骤 | 更新目标 | 更新内容 | 触发条件 |
|------|----------|----------|----------|
| 5.1 | 01_PROJECTS/{name}/STATUS.md | 项目阶段状态 | 涉及项目状态变更时更新 |
| 5.2 | 04_MEMORY/project/proj_{name}.md | 项目动态事实 | 项目状态变化时更新 |
| 5.3 | 06_LOGS/{project_name}/inspection/{YYYYMMDD}_project.md | 项目级巡检日志 | 每次巡检必写入 |

## 6. 巡检报告输出格式

Agent 完成阶段二至三后，必须生成巡检报告并弹窗展示给用户。报告格式如下：

```markdown
# AOS 项目级巡检报告 — {YYYY-MM-DD}

## 巡检概况
| 项目 | 值 |
|------|-----|
| 巡检时间 | {YYYY-MM-DD HH:MM} |
| AOS 版本 | {version} |
| 巡检类型 | 项目级 |
| 项目名称 | {project_name} |
| 目标机器 | {laptop/服务器} |
| 系统级巡检状态 | {已完成/未完成} |

## 项目状态
| 检查项 | 结果 | 备注 |
|--------|------|------|
| 项目配置 | PASS/FAIL | ... |
| 项目状态 | PASS/FAIL | ... |
| 项目记忆 | PASS/FAIL | ... |
| 双机同步 | PASS/FAIL/SKIP | ... |

## 文件分类结果

### A 类：完全无用的冗余文件（建议删除）
| 文件路径 | 类型 | 大小 | 删除原因 |
|----------|------|------|----------|

### B 类：内容过时需整合（建议修复/归档）
| 文件路径 | 问题 | 建议处置 |
|----------|------|----------|

### C 类：需留存的文件（无需操作）
| 文件路径 | 留存原因 |
|----------|----------|

### D 类：路径不合理需迁移（建议迁移）
| 文件路径 | 当前路径 | 建议路径 | 迁移原因 |
|----------|----------|----------|----------|

## 处置建议汇总
| 类别 | 文件数 | 建议操作 | 需用户确认 |
|------|--------|----------|------------|
| A | N | 删除 | 是 |
| B | N | 修复/归档 | 是 |
| C | N | 保留 | 否 |
| D | N | 迁移 | 是 |

## 待用户决策项
1. {决策项1}
```

## 7. 异常处理

| 异常场景 | 处理方式 |
|----------|----------|
| 服务器不可达 | 跳过双机检查，在报告中标注"服务器不可达，双机检查已跳过" |
| 发现关键状态文件损坏 | 立即停止巡检，弹窗警告用户，等待人工干预 |
| 用户拒绝处置建议 | 保留文件原状，在日志中记录"用户拒绝"，继续后续巡检 |
| 删除操作误删重要文件 | 立即停止，从 99_ARCHIVE/ 恢复（如已归档），报告用户 |
| LS 工具返回异常结果 | 改用 `Test-Path` / `Get-ChildItem -Force` 验证，不依赖 LS 工具结论 |
| 项目未完成初始化 | 提示用户补建缺失文件（AGENTS.md/README.md/STATUS.md） |

## 8. 巡检频率与触发机制

| 触发方式 | 频率 | 说明 |
|----------|------|------|
| 用户显式触发 | 每项目每日1次 | 用户发起项目工作会话时，Agent 自动执行前置巡检 |
| Agent 主动建议 | 检测到项目状态异常时 | Agent 发现项目状态不一致时建议用户发起巡检 |

**重要说明**：由于 AOS 是单次对话触发式 Agent，不存在后台常驻进程，因此"每日自动巡检"在技术上是不可行的。巡检必须由用户显式触发。本规范属于 Layer 3（用户指令显式触发），不属于 Layer 1（自动触发）。

---

## 9. 多项目场景处理

| 场景 | 处理方式 |
|------|----------|
| 系统级巡检完成后，发起项目A工作会话 | 执行项目A项目级巡检 → 开始项目A工作 |
| 同日发起项目B工作会话 | 检查系统级日志(已完成) → 执行项目B项目级巡检 → 开始项目B工作 |
| 同日再次发起项目A工作会话 | 检查项目A项目级日志(已完成) → 直接开始项目A工作 |
| 未执行系统级巡检就发起项目工作 | 提示用户"建议先执行系统级巡检"，用户可选择继续或先执行 |

## 10. 与现有规范的集成
